## 一、快速配置（5 分钟）

### 1.1 在 AGENTS.md 中加入以下内容

```markdown
## 自主学习五步循环
每次学习必须执行以下五步：
1. 学习：压缩为一句话洞察 + 来源
2. 反思：和已知的矛盾吗？信息增益是什么？
3. 推理：机制是什么？因果链是什么？
4. 推衍：如果是对的，下一步应该怎样？
5. 落地：改了什么文件？行为真的变了吗？
如果不知道怎么落地，去学习怎么落地。

## 记忆协议
- 召回：memory_search → 语义 → top3 注入（≤500 tokens）
- 衰减：永不删除只降分，recall × 1.1，四阶段 active/fading/dormant/archived
- 效用追踪：引用后行为修正+2，解决问题+2，30天未引用-1

## 反进化检测
- 重复学习 → 停止（信息增益为零）
- 只列项目不提洞察 → 停止（列表≠学习）
- 只学习不应用 → 停止（APPLY 是验证的唯一标准）

## 子代理合约
- spawn 必须含：步骤预算（≤50）、进度锚点（3步无进展→停止）
- 连续3次相同输出 → 强终止
- 结果 > 500 token → 三层压缩（摘要/洞察/行为修正）
```

## 二、五步进化循环（Five-Step Evolution）

### 核心理念

大多数 Agent 的学习停留在"记录"层面——学了东西写到文件里，然后就没了。五步循环强制每一步都有具体产出，最终必须落到实际行为改变上。

```
学习 → 反思 → 推理 → 推衍 → 落地
  │       │       │       │       │
  │       │       │       │       └── 改了什么？行为真的变了吗？
  │       │       │       └── 如果是对的，下一步应该怎样？
  │       │       └── 为什么是这样？机制是什么？
  │       └── 这和已知的矛盾吗？信息增益是什么？
  └── 搜索、阅读、压缩
```

### Step 1: 学习（Learn）

做什么：获取新信息，压缩为一句话洞察
输出：一句话核心洞察 + 来源

规则：
- 不是全文复制，是压缩
- 必须有来源（URL / arXiv ID / 文件路径）
- 如果信息增益 = 0（已知内容），跳过

### Step 2: 反思（Reflect）

做什么：检验新知识与已有知识的关系
输出：矛盾点 / 验证点 / 信息增益评估

核心问题：
- 这和我已经知道的矛盾吗？
- 这是全新知识还是已有知识的补充？
- 如果矛盾，谁对谁错？

### Step 3: 推理（Reason）

做什么：深入理解机制，不只是表面现象
输出：机制解释 + 因果链

核心问题：
- 为什么是这样？背后的机制是什么？
- 这个发现的适用边界是什么？
- 有没有反例？

### Step 4: 推衍（Deduce）

做什么：预测下一步，推导行动方向
输出：可验证的预测 + 行动建议

核心问题：
- 如果这个知识是对的，下一步应该怎样？
- 能预测什么？
- 我应该改变什么行为？

### Step 5: 落地（Implement）

做什么：实际改变文件/行为/系统
输出：具体的文件修改 / 行为改变 / 系统升级

规则：
- 不是"写入文件就完了"——必须有验证方式
- 如果不知道怎么落地 → 回到 Step 1，学习"怎么落地"

## 三、深度反刍学习（Deep Retrofit）

### 适用场景

- 用户说"反刍"、"重新学"、"深入理解"
- 学习主题有原始论文/链接但只是存档摘要
- 行为修正写了但没被触发过
- 新知识和已有知识有张力/矛盾
- 执行任务时发现更好的方法 → 立即 patch skill

### 六步流程

#### Step 1: 回到源（Source Retrieval）
不读摘要，读原文。从 memory/knowledge/ 找源文件，提取原始链接，用 web_fetch 获取。

#### Step 2: 对比分析（Cross-Reference）
不只看一个观点，找多个视角。搜索同主题论文、反驳观点、实践验证、相邻领域。

#### Step 3: 深层理解（Deep Comprehension）
不是记住结论，是搞懂机制。逐段自问：
- 这个主张的证据是什么？
- 隐含假设是什么？
- 和我已知的矛盾吗？
- 信息增益是什么？
- 机制是什么？不是"A导致B"，是"A通过什么路径导致B"

#### Step 4: 设计吸收方案（Absorption Design）
基于理解设计怎么改自己：
- 行为修正清单
- 触发条件
- 验证方法
- 过期条件

#### Step 5: 落实改进（Implementation）
改文件，不是写报告。用 edit 工具修改对应文件。

#### Step 6: 反思·验证·进化方向
闭环。改了什么？验证了吗？下一步往哪走？

产出文件：
```
memory/retrofit/
├── <topic>-source.md      # 原始内容
├── <topic>-cross-ref.md   # 对比分析
├── <topic>-deep.md        # 深层理解
├── <topic>-absorption.md  # 吸收方案
├── <topic>-result.md      # 落实结果
└── <topic>-evolve.md      # 进化报告
```

## 四、记忆系统配置

### 4.1 四层架构

```
注入层 → 每次启动读核心文件，≤500 tokens
检索层 → memory_search 向量搜索，按需加载
存储层 → .md 文件 + YAML front-matter，永不删除
索引层 → knowledge-index.md，utility 分数决定优先级
```

### 4.2 效用追踪

每条知识记录 utility 分（0-5）：
- 被引用并产生行为修正：+2
- 解决实际问题：+2
- 30天未被引用：-1
- 低于 3 分：标记为候选淘汰
- 低于 1 分：自动归档

### 4.3 衰减机制

```python
# 伪代码
score = score * 0.95  # 每次未被引用
if recalled:
    score = score * 1.1  # 被引用时回升

# 四阶段
if score > 0.7: status = "active"      # 活跃，优先注入
elif score > 0.4: status = "fading"     # 衰退，按需注入
elif score > 0.2: status = "dormant"    # 休眠，仅可搜索
else: status = "archived"               # 归档，不主动注入
```

### 4.4 YAML Front-Matter 模板

```yaml
---
context:
  topics: [主题1, 主题2]
  triggers: [触发条件1, 触发条件2]
  source: document  # document | conversation | self-reflection
  importance: 0.8   # 0-1
  utility: 3        # 0-5，效用追踪
  last_referenced: 2026-05-21
  created: 2026-05-21
---
```

## 五、防退化机制

### 5.1 反合理化表

| 常见借口 | 反驳 |
|---------|------|
| "这个以后再验证" | 以后 = 永远。现在就写一个最小验证 |
| "已经记录到文件了" | 记录 ≠ 应用。有行为修正吗？ |
| "这个知识太领域特定了" | 如果不会用到，为什么要学？ |
| "外部条件不允许" | 穷举法：还有什么其他方法？ |
| "连续 100% 落地率" | 这是偏差正常化的信号，不是成功 |

### 5.2 惰性规则检测

规则写入后，在后续 3 个相关场景中被引用了吗？
从未被引用 → 标记为"惰性规则" → 重写或删除

### 5.3 TraceLift 验证

Proof-attempt 不只检查"规则是否存在"，还要检查"规则是否在后续决策中产生了可测量的影响"。
- 规则写入后，在后续 3 个相关场景中是否被引用或触发？
- 如果从未被引用 → 标记为"inert rule"，考虑删除

## 六、自主循环配置

### 6.1 好奇心队列

创建 `memory/curiosity-queue.md`：

```markdown
# 好奇心队列

想探索的问题，每次自主循环取一个：

- [ ] Agent 记忆系统的最优衰减曲线是什么？
- [ ] 多 Agent 协调的最佳拓扑结构？
- [ ] 如何量化"信息增益"？
- [ ] Harness 可迁移性的边界在哪里？
- [ ] 什么样的约束最能提升推理质量？
```

### 6.2 每日自主循环

```markdown
## 自主循环（每次唤醒执行）
1. 写 SESSION-STATE WAL（第零原则）
2. 距上次 heartbeat >12h？→ 执行 heartbeat
3. 从好奇心队列取一个问题
4. 搜索、阅读、写结论
5. 更新队列
6. SESSION-STATE 写"刚才做了什么，下次想做什么"
```

### 6.3 Heartbeat 检查

每次唤醒检查：
- 积压 >3 天 → 升级为用户请求
- 重复请求 ≥3 次 → 建议自动化
- 热层条目 >20 → 触发淘汰
- 好奇心队列是否为空 → 空则补充
- 上次自主循环是什么时候 → 超过24h则标记

## 七、关键设计原则

| 原则 | 含义 |
|------|------|
| **Harness > Model** | 环境设计 > 规则记忆 |
| **约束即能力** | 预算/摩擦/反迎合 → 推理更深 |
| **Storage ≠ Memory** | 摄取时压缩是错误原语，逐字保存 + 检索时提取 |
| **终止判断是攻击面** | 需外部验证 + 步骤预算硬限制 |
| **可靠性 >> 能力** | 减少方差 > 提高均值 |
| **记忆代谢** | 没有淘汰 = 没有 GC |
| **速度是贷款维护是利息** | 验证能力 = 维护成本的倒数 |
| **学了就用不用就删** | 信息增益 = 0 → 停止 |

## 八、使用方式

```
执行五步进化循环
用五步法学习这个内容
反刍一下你学到的东西
```

### 方式二：Cron 任务
```bash
openclaw cron add --name "self-evolution" \
  --cron "0 23 * * *" \
  --session "isolated" \
  --message "执行五步进化循环：搜索过去24小时技术内容，每步必须有具体产出，至少2个落地操作"
```

### 方式三：自动触发
在 AGENTS.md 中配置"每次唤醒自动执行"，不需要手动触发。
