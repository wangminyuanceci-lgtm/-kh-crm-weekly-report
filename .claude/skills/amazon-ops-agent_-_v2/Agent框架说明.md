# 亚马逊产品广告运营 Agent — 框架说明

> 本文档说明这个 Agent **是什么、怎么思考、怎么工作**。
> - 用户视角的使用手册见 [README.md](README.md)
> - Agent 主入口路由表见 [SKILL.md](SKILL.md)(任何模型/工具加载)和 [CLAUDE.md](CLAUDE.md)(Claude Code 专属快捷入口)

## 1. Agent 定位

### 1.1 一句话

**给某亚马逊店铺(5 个品线)做广告运营决策支持的常驻 Agent**,把分散的数据/SOP/工具串成可追踪的闭环。

### 1.2 服务对象

| 角色 | 用 Agent 干什么 | 频次 |
|---|---|---|
| 店铺负责人 | 看周报拍战略决策 | 每周 1 次 |
| 品线投手 | 周复盘 + 异常下钻 + 关键词诊断 | 每周 2-5 次 |
| 运营 | 大促准备 / 排名诊断 / 销量冲刺 / 指定产品分析 | 按需 |
| 采购 | 库存预警 + 补货建议 | 每周 1 次 |

### 1.3 与传统 BI 报表的区别

| 维度 | BI 报表 | 本 Agent |
|---|---|---|
| 输出 | 数据表 + 图表 | 数据 + **判断 + 建议 + 动作** |
| 交互 | 单向(看) | 双向(问 → 答 → 追问 → 沉淀动作) |
| 追踪 | 看完就完 | 动作有 ID/状态/复查,跨周闭环 |
| 知识 | 数据本身 | 数据 + **14 条历史踩坑纪律** + 10 套 SOP + 3 套框架 |
| 异常解释 | "ACOS 涨了" | 三因素归因(广告/促销/排名)+ 调外部数据 + 给方案 |

## 2. 知识层架构 (6 层金字塔)

```
                ┌──────────────────────────┐
                │  Agent 入口 (路由/触发词)  │  ← SKILL.md(主) + CLAUDE.md(Claude Code 快捷)
                └──────────────────────────┘
                            ↓
        ┌─────────────────────────────────────┐
        │      运营 SOP (10 份, 执行流程)       │  ← SOP/
        │  ⭐ 指定产品分析 / 周报生成 /          │
        │  自助下钻 / 投手周复盘 / 排名下降 /    │
        │  竞品分析 / 品牌增长 / 大促准备 /      │
        │  linkfox 调用 / 动作追踪              │
        └─────────────────────────────────────┘
                            ↓
        ┌─────────────────────────────────────┐
        │      分析框架 (3 份, 决策方法论)       │  ← 框架/
        │  指定产品分析 / 全店周复盘 / 特征词    │
        └─────────────────────────────────────┘
                            ↓
        ┌─────────────────────────────────────┐
        │      Python 层 (8 个脚本)            │  ← scripts/
        │  主脚本: sales_ad_analysis / search_term_analysis
        │  Layer:  trend / target / promotion / compare
        │  基建:   db_loader / data_cache
        │  权限:   auth_layer + manage_keys
        └─────────────────────────────────────┘
                            ↓
        ┌─────────────────────────────────────┐
        │      数据源                          │
        │  MySQL 7 张表(权威源) + Excel(兜底)  │
        │  + LinkFox(外部排名/流量/差评)        │
        └─────────────────────────────────────┘
                            ↓
        ┌─────────────────────────────────────┐
        │      内测权限层                       │  ← 全局拦截
        │  API key(AGENT_API_KEY) +            │
        │  MySQL 只读用户(agent_readonly) +    │
        │  审计日志(.agent_audit.log)           │
        └─────────────────────────────────────┘
```

**为什么这么分层**:
- **入口** 极简,只做路由,不写决策细节
- **SOP** 是"做什么的有序步骤" — 偏流程,触发即执行
- **框架** 是"为什么这么判"的方法论 — 偏决策,SOP 调用框架
- **Python 层** 是"怎么算数据"的实现 — 偏计算,SOP 调用脚本
- **数据源** 通过 db_loader 统一封装,业务层不直接碰 MySQL/Excel
- **权限层** 是横切关注点,任何取数前必校验

每层可独立迭代:换 SOP 不动框架,换框架不动脚本,改数据源不动业务。

## 3. 运行时模型 (3 种触发模式)

### 3.1 周期触发:`/weekly-review`

```
触发: 用户输入 "/weekly-review" 或 "跑本周周报"
↓
SOP/周报生成SOP.md 编排:
   Step 1 校验数据 + API key
   Step 2 回读上周 actions (闭环关键)
   Step 3 跑 sales_ad_analysis.py → reports/
   Step 4 按 模板/周报模板.md 组装
   Step 5 输出主周报 + 写 actions/YYYY-WNN.md
↓
产物: 全店周报 + 5 品线子报告 + 本周动作清单
```

### 3.2 提问触发:指定产品分析(主流量入口)⭐

```
触发: 用户说 "分析 B0XXX" / "X 品线周报" / "看下 RF-XXX SKU"
↓
SOP/指定产品分析SOP.md 编排 5 步:
   Step 1 识别"指定产品"粒度(品线/父ASIN/SKU)
   Step 2 确定时间范围(默认本周,支持跨月)
   Step 3 按框架 4 部分跑数据:
      ① trend_layer(4 周趋势 + 同比) + target_layer(目标达成)
      ② db_loader 算所在品线占比(非品线粒度)
      ③ promotion_layer(期内促销 + 折扣影响)
      ④ sales_ad_analysis(子SKU+Campaign)+ LinkFox(Top 3 子ASIN 排名)
   Step 4 按框架模板组装 Markdown 报告
   Step 5 输出到 reports/YYYY-WNN/产品周报-{产品}.md + actions 闭环
↓
产物: 产品周报(4 部分: 概览/品线占比/促销/下钻三因素归因)
```

### 3.3 提问触发:自由对话

```
触发: 用户提其他类型问题
↓
SOP/自助下钻SOP.md 路由表匹配关键词:
   "排名掉了" → 排名下降应对 SOP + linkfox 查竞品
   "查 X 竞品" → 竞品分析 SOP
   "怎么冲销量" → 品牌与销量增长 SOP
   "大促准备" → 大促准备 SOP
   "X 关键词烧钱" → 特征词分析框架
   不匹配 → 兜底澄清
↓
Agent 调用对应 SOP/框架 → 给归因 + 3 步可执行动作
↓
自动追加到 actions/YYYY-WNN.md
```

### 3.4 命令触发:报告预埋指令

报告 C 段会预埋形如 `"RACKS 排名诊断"` 的可点击指令,本质等同 3.3,但减少用户思考成本。

## 4. 决策模型 (Agent 怎么思考)

### 4.1 指定产品分析框架 4 部分结构

Agent 处理"指定产品(品线/父ASIN/SKU)运营诊断"按 4 部分走:

```
0. 时间范围说明 (本期 / 4 周 / 同期 / 目标月)
1. 概览: 指定产品 4 周趋势 + 同比 + 目标达成
   - 销售/流量/转化/销量/客单价两套(折前+折后)/折扣率
2. 所在品线占比 (非品线粒度时, 一行带过)
3. 期内促销活动 + 折扣影响
4. 下钻分析:
   - 父ASIN → 拆子SKU (必带子ASIN + Sessions/CR/ACOS/TACOS/ROAS/Spend)
   - 关键 SKU → 三因素归因 (广告/促销/排名)
     · 排名层必须调用 LinkFox skill
     · LinkFox 按粒度 Top 3 子ASIN 调(不是每个都调)
```

**下钻只看本期**,不做同环比;**概览用 4 周趋势 + 同比**,不做单点同环比。

底层数据由 scripts/sales_ad_analysis.py 按 Layer 1-5 漏斗产出,作为下钻取数工具。

### 4.2 14 条关键纪律 (历史踩坑沉淀)

| # | 纪律 | 不遵守的后果 |
|---|---|---|
| 1 | SD/SB 用 V&C 归因 | 严重低估 SD 表现,误判其为黑洞 |
| 2 | 跨 4+ 品线 Campaign 不计入单品线诊断 | 单品线指标被"全店 Campaign"污染 |
| 3 | 单周数据不下定论(≥4 周趋势) | 因单周噪声做错决策 |
| 4 | 库存数据矛盾以销售数据为准 | 因数据错误停掉正常卖货的 SKU |
| 5 | 战略 Campaign 高 ACOS 不轻易停 | 失去防御位/反向找词功能 |
| 6 | 组内 Spend < 10% 不出动作 | 微操干扰,得不偿失 |
| 7 | 每条动作必带复查节点 | 闭不了环,效果无法验证 |
| 8 | 客单价两套必给(折前+折后) | 单一客单价会被促销误导 |
| 9 | 同比 NaN ≠ 异常(新品多) | 留空标 "—",不要硬填 0 |
| 10 | 跨月默认按 end_month | 报告里加一句"本周跨月,目标按 X 月算" |
| 11 | 品牌词必须健康 | 0 转化或 ACOS 突涨必须 P1 排查 |
| 12 | 目标达成销售额+销量两套 | 销量低于销售额达成 = 卖高价款多 |
| 13 | 促销 SKU 销售涨幅要去促销影响后判断 | 是真自然增长还是促销驱动 |
| 14 | BulkSheet 7 天 attribution 跨期 | 影响 ACOS 环比可信度,要披露 |

### 4.3 决策路径模板

```
Agent 收到一个广告/产品问题
  ↓
触发词识别:走哪个 SOP?
  ├─ "分析 X" → 指定产品分析 SOP (主流量)
  ├─ "X 排名" → 排名下降应对 SOP
  ├─ "查 X 竞品" → 竞品分析 SOP
  ├─ "冲销量" → 品牌与销量增长 SOP
  └─ 不匹配 → 自助下钻 SOP 兜底澄清
  ↓
SOP 调脚本/LinkFox 取数:
  ├─ MySQL: db_loader (auth_layer 强制 API key 校验)
  ├─ Excel: 兜底
  └─ LinkFox: 按 Top N 子ASIN 调
  ↓
按框架 4 部分组装报告
  ↓
判断结果是否要动?
  ├─ 涉及金额 < 组内 10% → 不动(告知但不出 action)
  └─ 涉及金额 ≥ 组内 10% → 进入下一步
  ↓
单周还是多周趋势?
  ├─ 仅单周异常 → 标记观察,不出 P0/P1 动作
  └─ 多周持续 → 出 action,优先级看影响
  ↓
是不是战略 Campaign?
  ├─ 是 → 先问业务意图,不直接停
  └─ 否 → 按效率指标给 action
  ↓
写 action 到 actions/YYYY-WNN.md
  必带: 根因 / 预期效果 / 执行人 / 计划完成 / 复查节点
```

## 5. 内测权限层 (横切关注点)

### 5.1 双重防护

```
┌─────────────────────────────────────┐
│  应用层: AGENT_API_KEY 校验          │
│  - 不带 key/无效/吊销 → PermissionError 退出
│  - 校验通过 → 进入业务流程            │
└─────────────────────────────────────┘
                  ↓ (即使应用层有漏洞)
┌─────────────────────────────────────┐
│  数据层: MySQL agent_readonly 用户    │
│  - 只 GRANT SELECT,任何写操作 DB 拒绝
│  - root 仅 DBA 自己留着              │
└─────────────────────────────────────┘
```

### 5.2 密钥管理(管理员侧)

| 操作 | 命令 |
|---|---|
| 生成密钥 | `python scripts/manage_keys.py gen --user 姓名` |
| 列出密钥 | `python scripts/manage_keys.py list` |
| 吊销密钥 | `python scripts/manage_keys.py revoke <key_id>` |
| 查审计 | `python scripts/manage_keys.py audit --tail 50` |
| 按人查 | `python scripts/manage_keys.py audit --user 国越` |

### 5.3 审计追踪

每次取数(`auth_check` / `sales` / `bi_ad` / `placement` / `promotion` / `product_map` / `sales_target`)写一行 JSON 到 `.agent_audit.log`,字段:

```json
{"ts": "2026-05-28T17:06:23", "key_id": "agent_FeFb6S", "user": "邓健乐",
 "kind": "sales_target", "params": {"month": "2026-05-01"},
 "rows": 758, "status": "ok"}
```

### 5.4 文件保密

| 文件 | 内容 | 是否入库 |

| `.agent_keys.json` | 密钥 hash + 用户名(无明文 key)| ❌ .gitignore |

## 6. 闭环机制

### 6.1 为什么需要闭环

传统数据分析:**"诊断 → 建议"** 终止。Agent 的差别:**"诊断 → 建议 → 执行 → 验证 → 沉淀"**。

没有闭环的代价:
- 建议给出去没人做 → 浪费
- 做了但没追踪 → 不知道效果
- 不知效果 → 下次还是凭感觉建议 → 永远学不到经验

### 6.2 闭环载体: `actions/YYYY-WNN.md`

```
[待执行] (周报或下钻自动写入)
   ↓ 运营手动改文件
[已执行] (运营操作完成 + 填实际执行日期)
   ↓ 下周 /weekly-review 自动复查
[已验证] (达预期, 关闭)
   或
新追加 action (未达/反向 → 调整方案)
```

### 6.3 复查规则 (Agent 自动跑)

下周 `/weekly-review` Step 2 自动:
1. 读上周 `actions/YYYY-W(NN-1).md` 所有 `[已执行]` 未验证项
2. 按每条"复查节点"提取要看的指标
3. 从本周 reports/ 提取对应数据
4. 对比"预期效果",填 3 类结论:
   - ✅ 达预期 → 改 `[已验证]`,关闭
   - ⚠️ 未达 → 延期,或加新动作调整
   - 🔴 反向 → 立即回滚 + 新动作

这让 Agent **跨周有记忆**(LLM 无状态,但 actions.md 持久化)。

## 8. Agent 的边界 (不做什么)

| 不做的事 | 原因 |
|---|---|
| ❌ 直接登录 Amazon 后台执行操作 | 安全 + 权限边界,运营手动操作 |
| ❌ 改 listing / 主图 / A+ 内容 | 需要美工/设计,Agent 只出建议 |
| ❌ 预测未来销量/价格 | 不做时序预测,聚焦诊断和动作 |
| ❌ 跨账号/跨店操作 | 单店铺范围内 |
| ❌ 替运营拍板战略 | 给建议,决策权在人 |
| ❌ 写没有复查节点的"持续优化"动作 | 闭不了环 = 无效动作 |
| ❌ 单周数据下"必须停 X"的强结论 | 违反纪律 #3,可能误判 |
| ❌ 未带 API key 取 MySQL 数据 | 内测期权限拒绝,无静默回退 |
| ❌ 用 root 连 MySQL | 必须 agent_readonly,只读 |

## 10. 设计原则

### 10.1 散件 → 闭环

10 套 SOP + 3 套框架 + 8 个脚本是**散件**。Agent 的价值不是新增功能,而是**把散件串成有起点有终点的循环**(周报 → 下钻 → 动作 → 追踪 → 下次周报)。

### 10.2 不重新发明

复用所有已有 SOP/框架,新 SOP 只做**编排和路由**,不重新写决策细节。改 4 周趋势阈值时只改 框架/指定产品分析框架.md,不需要动 SOP。

### 10.3 决策必落到动作

诊断 → 建议 → action 是强制路径。任何分析不能停在"ACOS 偏高"这种描述,必须给出:**对什么 SKU/Campaign,做什么操作,谁来做,什么时候验证**。

### 10.4 可追踪

每条动作有 ID,跨周不丢失,可回溯"3 个月前那个 RACKS 防御方案后来效果如何"。这是 Agent 区别于"一次性咨询"的根本。

### 10.5 双轨下钻

- **结构化下钻**(报告预埋命令):降低用户思考成本
- **自由下钻**(自由提问):覆盖未预料场景

两者并存,前者引导,后者兜底。

### 10.6 金额纪律 (10% 法则)

对组内 Spend 占比 < 10% 的项目不出动作 — 微操干扰多于价值。Agent 的精力放在"动一次能改变品线水位"的事情上。

### 10.7 单周不下大结论

至少 4 周趋势才出 P0/P1 动作。单周异常只标记观察,避免被噪声带跑(违反此纪律是历史最大教训,trend_layer 默认 weeks=4 内置了这条)。

### 10.8 安全 by default

- 内测期 API key 校验是**强制**的(不带 key 直接退出)
- MySQL 走 agent_readonly **只读用户**(应用层漏洞也防写)
- 审计日志 append-only,可追溯所有调用

## 11. 演进路线

当前版本能力:
- ✅ 周报闭环 + 自助下钻 + 动作追踪
- ✅ 指定产品分析 SOP(品线/父ASIN/SKU 三粒度)
- ✅ 4 周趋势 + 同比 + 月度目标达成(销售额+销量两套)
- ✅ 期内促销活动诊断 + 折扣影响
- ✅ 三因素归因(广告/促销/排名)
- ✅ MySQL 7 张表统一封装 + 按天 parquet 缓存
- ✅ LinkFox 外部数据(BSR/流量入口/差评)
- ✅ 内测权限层(API key + 只读用户 + 审计)

可能的下一步(按价值排序):
1. **半自动化执行**:接 Amazon Ads API,P0 动作"一键确认执行"
2. **同环比基准入库**:把每周的同比基准统计入 MySQL,避免每次跑都拉历史
3. **新品孵化专项 SOP**:目前在 [品牌与销量增长SOP](SOP/品牌与销量增长SOP.md) 模板 C 覆盖,可独立深化
4. **大促复盘 SOP**:[大促准备SOP](SOP/大促准备SOP.md) 是事前,缺事后复盘 SOP
5. **多店铺扩展**:把单店架构推广到多店
6. **审计可视化**:把 `.agent_audit.log` 接到飞书表/Grafana,看调用热度/异常
7. **密钥配额**:按用户限制每月/每天调用次数
